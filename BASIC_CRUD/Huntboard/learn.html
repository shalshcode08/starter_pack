Huntboard, explained from the metal up

Everything built in Phase 1 (foundation) and Phase 2 (data model and auth), explained so you can rebuild it yourself and defend every line in an interview.

How to read this

Each section answers three questions: what the thing is, why we chose it over alternatives, and how the code actually does it. Watch for the three callout types:

? Why the reasoning behind a decision   ! Gotcha a trap or version-specific surprise   ⚑ Interview a point worth being able to talk about.

If you only have five minutes: read section 1 (the big picture), section 11 (the data model), and section 18 (the flows). Those three carry the spine of the app.

1. The big picture top

Huntboard is a job-application tracker. A user signs up, then records job applications, each moving through a pipeline of stages and carrying a timeline of events. Everything a user sees is strictly their own data.

The whole thing is one Next.js application that runs on a server (Node.js) and talks to a PostgreSQL database hosted on Neon. There is no separate backend service - in the App Router, your server code (database queries, auth, validation) and your UI live in the same project and are split apart at build time.

  Browser                Next.js server (Node)              Neon (PostgreSQL)
  -------                ---------------------              -----------------
  request  ───────────▶  proxy.ts  (is user logged in?)
                            │ no ──▶ redirect to /login
                            │ yes
                            ▼
                         Server Component / Server Action
                            │  uses prisma (lib/prisma.ts)
                            │  ───────────────────────────────▶  SQL query
                            │  ◀───────────────────────────────  rows
                            ▼
                         HTML / response  ──────────────────▶  Browser renders

Two phases are done:

  • Phase 1 - Foundation: the Next.js app, styling (Tailwind), the database connection through Prisma, and a deployable shell.
  • Phase 2 - Data and auth: the database tables (User, Application, Event), and a full email/password login system with protected pages.
Interview framing

"It is a full-stack Next.js app with server-rendered pages, a Postgres database accessed through Prisma, and self-hosted credentials auth with JWT sessions. Every query is scoped to the authenticated user." That one sentence signals you understand the whole system.

3. The App Router mental model top

This is the single most important concept. Next.js App Router code runs in two different places, and knowing which is which explains everything else.

Server Components (the default)

Every file in app/ is a Server Component unless it says otherwise. It runs only on the server, never ships to the browser, and can do server things directly: read the database, read secrets, check the session. Our app/(app)/board/page.tsx is a server component - notice it calls await auth() right inside the component.

Client Components

A file that starts with "use client" is a Client Component. It is sent to the browser and can use interactivity: state, effects, event handlers, form hooks. Our components/auth/auth-form.tsx is a client component because it uses useActionState and useFormStatus.

Server Actions

A function marked "use server" is a Server Action: a server function you can call from a form or a client component as if it were local, but it actually runs on the server. This is how our login and signup forms talk to the database without us writing a single API route by hand. (More in section 17.)

Proxy (formerly "middleware")

The proxy.ts file runs before any page, on every matching request. We use it to check the session and redirect logged-out users away from protected pages. (More in section 15.)

Gotcha

In Next.js 16 the file convention was renamed from middleware.ts to proxy.ts, and the exported function must be a real function (default export or named proxy), not a destructured const. We hit this exact error and fixed it.

5. pnpm and the package files top

pnpm is the package manager. It installs dependencies into a single global content-addressed store and links them into each project, so the same package is never downloaded or stored twice. It is also strict: a package can only import what it actually declares, which prevents "it works by accident" bugs.

package.json - the scripts that matter

"scripts": {
  "dev": "next dev",                       # local dev server
  "build": "prisma generate && next build", # generate client, then build
  "start": "next start",                   # run the production build
  "postinstall": "prisma generate",        # regenerate client after install
  "db:migrate": "prisma migrate dev",       # create + apply a migration (dev)
  "db:deploy": "prisma migrate deploy",     # apply migrations (prod)
  "db:studio": "prisma studio"              # visual DB browser
}
Why prisma generate in build and postinstall

The Prisma client is generated code we keep out of git (see section 9). So anywhere the code is freshly installed (your machine, or Vercel's build), we must regenerate it before the TypeScript compiles, or the import @/generated/prisma/client would not exist.

pnpm-workspace.yaml - allowed build scripts

pnpm does not run a dependency's install scripts by default (a supply-chain safety measure). Some packages genuinely need theirs. We explicitly allow them:

allowBuilds:
  '@prisma/engines': true   # downloads the query engine binary
  prisma: true
  sharp: true
  unrs-resolver: true

pnpm-lock.yaml

The lockfile pins the exact resolved version of every package (direct and transitive). Commit it. It is what makes installs reproducible across machines and CI.

6. TypeScript config top

Two lines in tsconfig.json matter most for understanding the codebase.

"strict": true,            # all strict checks on: no implicit any, null checks, etc.
"paths": { "@/*": ["./*"] } # import alias: @/lib/prisma == ./lib/prisma

The @/ alias means imports are written from the project root instead of with fragile relative paths like ../../../lib/prisma. So @/lib/prisma, @/auth, and @/generated/prisma/client all resolve from the root.

Gotcha

Because our app uses the root layout (no src/ folder), the alias maps @/* to ./*. If you had used a src/ directory it would map to ./src/*. Mixing them up is a classic "module not found" head-scratcher.

7. Environment, secrets, and Neon branching top

Two files, one rule

  • .env - holds the real secrets (database URL, auth secret). Gitignored. Never committed.
  • .env.example - a committed template with placeholder values, so anyone cloning the repo knows which variables to set.

Our .gitignore ignores all .env* files but re-includes the example with !.env.example.

DATABASE_URL="postgresql://...neon.tech/neondb?sslmode=require&channel_binding=require"
AUTH_SECRET="...generated with: openssl rand -base64 33..."

AUTH_SECRET is the key NextAuth uses to sign and encrypt the session token. If it leaks or changes, existing sessions become invalid.

Neon branching: separate dev and prod databases

We deliberately do not share one database between development and production. Neon's branching gives us two isolated Postgres databases inside one project:

  Neon project "huntboard"
   ├─ production branch   →  ep-red-waterfall...   (Vercel uses this)
   └─ development branch  →  ep-small-bird...      (your local .env uses this)
Why separate

If dev and prod shared a database, a test record or a half-applied migration during development could corrupt real user data. Isolated branches mean you can experiment freely. Each branch is a full, independent Postgres database with its own connection string.

Gotcha: pooled vs direct connection

Neon gives a "pooled" string (host contains -pooler) and a "direct" one. Connection pooling (PgBouncer) is great for serverless app runtime, but it breaks schema migrations. For local dev and migrations we use the direct string. We will reintroduce the pooled string for the deployed runtime in the deploy phase.

8. Prisma and the driver adapter top

Prisma is the ORM (Object-Relational Mapper). You describe your tables in prisma/schema.prisma, Prisma generates a fully typed client, and you query the database in TypeScript instead of raw SQL.

The three config pieces

schema.prisma (top of file)

generator client {
  provider = "prisma-client"      # Prisma 7's new generator (ESM, custom output)
  output   = "../generated/prisma" # where the client code is written
}
datasource db {
  provider = "postgresql"          # which database engine
}

prisma.config.ts

New in Prisma 7. It loads .env and tells the Prisma CLI where the schema, migrations, and connection URL are.

import "dotenv/config"; # loads .env into process.env
export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: { path: "prisma/migrations" },
  datasource: { url: process.env["DATABASE_URL"] },
});
Prisma 7 breaking change #1

The connection URL no longer lives in the datasource block of schema.prisma. It moved to prisma.config.ts, which loads it from the environment via dotenv. Older tutorials still show url = env("DATABASE_URL") in the schema.

The driver adapter

Prisma 7's client is "engine-less": instead of bundling a query engine that opens its own DB connection, you hand it a database driver through an adapter. We use @prisma/adapter-pg, which wraps pg (node-postgres) over a normal TCP connection.

Prisma 7 breaking change #2

new PrismaClient() now requires an adapter argument. Calling it with no arguments is a type error ("Expected 1 arguments, but got 0"). We hit this during the Phase 1 build and fixed it by passing the pg adapter.

Why adapter-pg and not adapter-neon

The Neon serverless adapter talks over WebSockets, which is built for edge runtimes. Our server code runs on the Node.js runtime, where a plain TCP connection via pg is simpler and more robust. Fewer moving parts, no WebSocket edge cases. If we ever needed edge, we would switch adapters - the rest of the code would not change.

9. The Prisma client singleton top

lib/prisma.ts

import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "@/generated/prisma/client";

const connectionString = process.env.DATABASE_URL;
if (!connectionString) {
  throw new Error("DATABASE_URL is not set");  # fail fast, with a clear message
}

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined };

const adapter = new PrismaPg({ connectionString });

export const prisma = globalForPrisma.prisma ?? new PrismaClient({ adapter });

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;  # reuse across hot reloads in dev
}
Why the global trick

In development, Next.js hot-reloads your code on every save. Without this pattern, each reload would create a new PrismaClient and open a new pool of DB connections, quickly exhausting the database's connection limit ("too many connections"). Stashing the client on globalThis means we reuse the same instance across reloads. In production there is no hot reload, so we skip the global and just create one.

Interview

This exact snippet is a known Next.js + Prisma pattern. Being able to explain why (hot reload connection exhaustion) is a small but real signal that you have actually run these tools, not just copied them.

Everywhere else in the app we simply import { prisma } from "@/lib/prisma" and call typed methods like prisma.user.findUnique(...).

10. Migrations top

A migration is a versioned SQL file that describes a change to the database schema. Running migrations in order takes any database from empty to the current shape. They are the database equivalent of git commits.

CommandWhat it doesWhen
prisma migrate devDiffs schema vs DB, writes a new SQL migration file, applies it, regenerates the clientLocal development
prisma migrate deployApplies existing migration files only (never generates)Production / CI
prisma db pushForce the DB to match the schema with no migration fileQuick prototyping only

We used db push once in Phase 1 just to prove the connection worked on an empty schema, then switched to real migrate dev in Phase 2 so we have a proper, versioned history (the file prisma/migrations/2026..._init/migration.sql).

Gotcha: the shadow database

migrate dev needs a temporary "shadow database" to detect drift. It creates and drops one automatically, which requires the DB user to have create-database permission. Neon's neondb_owner has it, so it just worked. On some managed Postgres providers you must configure a separate shadow DB URL.

11. The data model top

The heart of the app. Three tables and two enums. Read this slowly - the relationships are what make it more than a CRUD demo.

prisma/schema.prisma

enum ApplicationStatus { WISHLIST  APPLIED  OA  INTERVIEW  OFFER  REJECTED }
enum EventType        { APPLIED  OA  INTERVIEW  FOLLOW_UP  OFFER  REJECTED  NOTE }

model User {
  id           String   @id @default(cuid())
  name         String?                  # ? means optional / nullable
  email        String   @unique      # no two users share an email
  passwordHash String                   # bcrypt hash, never the raw password
  isGuest      Boolean  @default(false)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt      # auto-set on every update
  applications Application[]            # one user has many applications
}

model Application {
  id          String            @id @default(cuid())
  userId      String                              # foreign key to User
  company     String
  role        String
  jobUrl      String?
  location    String?
  salaryRange String?
  status      ApplicationStatus @default(WISHLIST)
  source      String?
  appliedAt   DateTime?
  createdAt   DateTime          @default(now())
  updatedAt   DateTime          @updatedAt
  user   User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  events Event[]                                  # one application has many events
  @@index([userId])                              # fast "all apps for a user"
}

model Event {
  id            String    @id @default(cuid())
  applicationId String
  type          EventType
  notes         String?
  scheduledAt   DateTime?
  createdAt     DateTime  @default(now())
  application Application @relation(fields: [applicationId], references: [id], onDelete: Cascade)
  @@index([applicationId])
}

Concept by concept

ThingWhat it means
@id @default(cuid())Primary key, auto-generated as a collision-resistant URL-safe string. Chosen over auto-increment integers, which leak how many rows exist and are guessable.
String?The ? makes a field optional (nullable). jobUrl, location, salaryRange may be empty.
@uniqueDatabase-enforced uniqueness. The signup flow relies on this to reject duplicate emails.
@relation(... onDelete: Cascade)Defines the foreign key. Cascade means deleting a User automatically deletes their Applications, and deleting an Application deletes its Events. No orphan rows.
User[] / Event[]The "many" side of a one-to-many relation. Prisma lets you traverse it in code: user.applications.
@@index([userId])A database index on the foreign key, so "fetch all applications for this user" stays fast as data grows.
@updatedAtPrisma sets this timestamp automatically on every update.
enumA fixed set of allowed values. status can only ever be one of the six pipeline stages - the database itself enforces it.
  User  ──(1 to many)──▶  Application  ──(1 to many)──▶  Event
  one person             many job apps                many timeline
                         (each has a status)          events per app
Interview gold

The one-to-many relations plus cascade deletes plus the enum-constrained pipeline are exactly what turns this from "a table of rows" into "a person who models relational data." When asked about the schema, lead with the relationships and why cascade deletes prevent orphans.

Why no OAuth / Session tables

We chose email + password only (no Google login) and JWT sessions. So we deliberately did not add the Account / Session / VerificationToken tables that NextAuth's database adapter would need. Carrying unused tables is dead weight. Lean schema = honest schema.

12. Authentication concepts top

Before the code, the vocabulary. Mixing these up is the most common auth confusion.

TermMeaning
AuthenticationProving who you are (logging in with email + password).
AuthorizationDeciding what you are allowed to do (only see your own applications).
SessionThe server's memory that "this browser is logged in as user X" across requests.
JWTJSON Web Token. A signed, tamper-proof token stored in a cookie that contains the session data, so the server does not need a session table.
HashA one-way scramble of the password. You can verify a guess against it but never reverse it back to the password.
CSRFCross-Site Request Forgery. An attack where another site tricks your browser into submitting a request. NextAuth defends with a token (section 18).

Why JWT sessions (not database sessions)

Two reasons

1. Required: NextAuth's Credentials provider only works with the JWT strategy, not database sessions. 2. Better anyway: JWT is stateless - the session lives in the signed cookie, so there is no database read on every request to look up a session. That scales more cleanly.

13. NextAuth v5 wiring top

The auth setup is intentionally split into two files. Understanding why it is split is the key insight.

The split: auth.config.ts vs auth.ts

Why two files

The proxy.ts (which runs on every request) should stay lightweight and must not import heavy, Node-only code like Prisma or bcrypt. So we put the lightweight, portable config (session strategy, callbacks, the "is this route allowed" check) in auth.config.ts, and the heavy part (the Credentials provider that touches the database) in auth.ts. The proxy imports only the lightweight half.

auth.config.ts (lightweight, imported by the proxy)

export const authConfig = {
  trustHost: true,
  session: { strategy: "jwt" },        # stateless sessions in a cookie
  pages: { signIn: "/login" },         # our custom login page
  providers: [],                          # filled in by auth.ts
  callbacks: {
    authorized({ auth, request: { nextUrl } }) {
      const isLoggedIn = !!auth?.user;
      const isAuthPage = ["/login", "/signup"].includes(nextUrl.pathname);
      if (isAuthPage) {                     # logged-in users skip login/signup
        if (isLoggedIn) return Response.redirect(new URL("/board", nextUrl));
        return true;
      }
      return isLoggedIn;                    # every other matched route requires login
    },
    jwt({ token, user }) {                # runs when the token is created/updated
      if (user) token.id = user.id;        # put the DB id into the token
      return token;
    },
    session({ session, token }) {         # shapes what the app sees as `session`
      if (token.id) session.user.id = token.id as string;
      return session;
    },
  },
};

auth.ts (full config, imports DB + bcrypt)

export const { handlers, auth, signIn, signOut } = NextAuth({
  ...authConfig,
  providers: [
    Credentials({
      async authorize(credentials) {
        const parsed = loginSchema.safeParse(credentials); # validate input
        if (!parsed.success) return null;
        const { email, password } = parsed.data;
        const user = await prisma.user.findUnique({ where: { email } });
        if (!user) return null;                          # unknown email
        const valid = await verifyPassword(password, user.passwordHash);
        if (!valid) return null;                          # wrong password
        return { id: user.id, email: user.email, name: user.name }; # success
      },
    }),
  ],
});

NextAuth(...) returns four things we use everywhere:

  • handlers - the GET/POST HTTP routes, re-exported in app/api/auth/[...nextauth]/route.ts.
  • auth - call it in a server component to read the current session.
  • signIn / signOut - call from server actions to start/end a session.

The authorize function is the trust boundary

Returning a user object = success. Returning null = rejected. Notice it returns the same null for "unknown email" and "wrong password" - we never tell an attacker which one was wrong.

How the id reaches the app

  authorize() returns { id, email, name }
        │
        ▼
  jwt callback:  token.id = user.id        (id baked into the signed cookie)
        │
        ▼
  session callback:  session.user.id = token.id
        │
        ▼
  any server component:  const s = await auth();  s.user.id   ← your user id
Gotcha: the type augmentation

By default TypeScript does not know session.user.id exists. We add it in types/next-auth.d.ts by augmenting NextAuth's Session interface. (The JWT side did not pick up cleanly in this beta, so we cast token.id as string in the session callback - a small, contained workaround.)

14. Password hashing top

lib/password.ts

import bcrypt from "bcryptjs";
const SALT_ROUNDS = 12;
export function hashPassword(password: string) { return bcrypt.hash(password, SALT_ROUNDS); }
export function verifyPassword(password: string, hash: string) { return bcrypt.compare(password, hash); }
  • We store only the hash, never the password. Even if the database leaks, passwords are not exposed.
  • Salt rounds = 12: bcrypt automatically generates a random salt and runs 2^12 iterations. Higher = slower = harder to brute force. 12 is a sane modern default.
  • The salt is stored inside the resulting hash string, which is why compare needs only the password and the hash.
Gotcha: the 72-byte limit

bcrypt only looks at the first 72 bytes of a password. Our Zod schema caps password length at 72 so a longer password is not silently truncated, which would be a subtle correctness bug.

15. Route protection (proxy.ts) top

proxy.ts

import NextAuth from "next-auth";
import { authConfig } from "@/auth.config";
const { auth } = NextAuth(authConfig);   # lightweight config only - no DB/bcrypt
export default auth;                       # the function Next runs before pages
export const config = {
  matcher: ["/((?!api/auth|_next/static|_next/image|favicon.ico|.*\\..*).*)"],
};

On every matching request, the proxy runs the authorized callback from auth.config.ts. If it returns false, NextAuth redirects to /login. The matcher is a regex that excludes the auth API, Next's internal assets, the favicon, and any file with an extension - so the proxy only guards real pages.

Why protect twice (proxy AND layout)

The proxy redirects logged-out users early. But our app/(app)/layout.tsx also calls await auth() and redirects if there is no session. This is defense in depth: even if a matcher rule was wrong, the page itself still refuses to render for a logged-out user. The layout also gives us the session object to show the user's email and a sign-out button.

app/(app)/layout.tsx (the guard part)

const session = await auth();
if (!session?.user) {
  redirect("/login");   # server-side redirect, runs before any UI
}

16. Validation with Zod top

Never trust input from the browser. Zod schemas validate and transform input at the server boundary, and give us TypeScript types for free.

lib/validations/auth.ts

const email = z.email("Enter a valid email")
  .transform((v) => v.toLowerCase().trim());   # normalize before storing

const password = z.string()
  .min(8, "Password must be at least 8 characters")
  .max(72, "Password must be at most 72 characters"); # bcrypt limit

export const loginSchema  = z.object({ email, password });
export const signupSchema = z.object({ name: z.string().trim().min(1).max(100), email, password });
  • safeParse(input) returns { success, data } without throwing - we check success and use the clean, typed data.
  • The email transform means " Me@X.com " is always stored and compared as "me@x.com", so casing/whitespace can never create duplicate-looking accounts.
  • z.infer<typeof signupSchema> derives the TypeScript type from the schema - one source of truth.
Interview

"Validate on the server, not just the client" is a security fundamental. Client validation is UX; server validation is the real gate. We do both.

17. Server Actions top

Server Actions let a form call a server function directly. No hand-written API route, no fetch, no JSON plumbing.

lib/actions/auth.ts (signup, condensed)

"use server";   # everything in this file runs only on the server

export async function signup(_prev, formData) {
  const parsed = signupSchema.safeParse({
    name: formData.get("name"),
    email: formData.get("email"),
    password: formData.get("password"),
  });
  if (!parsed.success) return { error: parsed.error.issues[0]?.message };

  const { name, email, password } = parsed.data;
  try {
    await prisma.user.create({
      data: { name, email, passwordHash: await hashPassword(password) },
    });
  } catch (error) {
    if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === "P2002")
      return { error: "An account with this email already exists" };  # unique violation
    throw error;
  }
  await signIn("credentials", { email, password, redirectTo: "/board" });
}

Key ideas:

  • Signature is (previousState, formData) because it is used with React's useActionState hook, which threads error state back into the form.
  • Postgres error code P2002 is a unique-constraint violation. We rely on the database's @unique on email rather than a "does this email exist" pre-check, which would have a race condition.
  • On success it calls signIn, which logs the new user in and redirects to /board.

The client side that calls it

The shared components/auth/auth-form.tsx is a client component using two hooks:

  • useActionState(action, undefined) - wires the server action to the form and exposes its returned state (so we can show state.error).
  • useFormStatus() - tells the submit button when the action is pending, so we disable it and show "Please wait...".
Gotcha: signIn throws a redirect

On success, signIn(... redirectTo) works by throwing a special redirect signal. So in a try/catch you must only catch real auth errors (AuthError) and re-throw everything else, or you would accidentally swallow the redirect. Our login/signup actions do exactly that.

18. End-to-end flows top

Sign up

1. User fills the form at /signup  (client component)
2. Form submit calls the `signup` server action with FormData
3. Zod validates + normalizes (lowercase email, length checks)
4. password is bcrypt-hashed (12 rounds)
5. prisma.user.create inserts the row
      └─ duplicate email? Postgres P2002 → "account already exists"
6. signIn("credentials") logs them in → JWT cookie set
7. throw redirect → browser lands on /board

Log in

1. User submits /login form → `login` server action
2. signIn("credentials") triggers authorize() in auth.ts
3. authorize: Zod parse → find user by email → bcrypt compare
      └─ any failure returns null → "Invalid email or password"
4. success → jwt callback bakes user.id into the signed token
5. session cookie set → redirect to /board

Visiting a protected page

1. GET /board
2. proxy.ts runs authorized() → reads JWT cookie
      └─ no valid session → 307 redirect to /login?callbackUrl=/board
3. session valid → (app)/layout.tsx ALSO calls auth() (defense in depth)
4. board/page.tsx (server component) calls auth() and renders

We verified this exact behavior with curl in Phase 2: /board unauthenticated returned 307 → /login, and after a real CSRF + credentials POST the session endpoint returned the user with their id, and /board returned 200.

The CSRF handshake (what that curl test exercised)

1. GET /api/auth/csrf      → returns a csrfToken + sets a cookie
2. POST /api/auth/callback/credentials
        body: csrfToken + email + password   (cookie sent back)
3. NextAuth checks the token matches the cookie  → blocks forged requests
4. authorize() runs → on success, session cookie returned
Why CSRF matters

Without it, a malicious page could auto-submit a login/logout request using your browser's cookies. The double-submit token (must appear in both the cookie and the body) proves the request came from our own form.

Sign out

The header's sign-out button is a tiny form whose action is the logout server action, which calls signOut({ redirectTo: "/login" }) - clearing the cookie and redirecting.

22. Self-test top

If you can answer these out loud, you own this codebase. Click to reveal.

Why is the Prisma client cached on globalThis in development? Next.js hot-reloads on every save. Without caching, each reload creates a new PrismaClient with a new connection pool, exhausting the database's connection limit. Caching on globalThis reuses one instance. Production skips this because there is no hot reload.
Why split auth into auth.config.ts and auth.ts? proxy.ts runs on every request and must stay lightweight, so it cannot import Node-only code like Prisma or bcrypt. The lightweight config (session strategy, callbacks, route check) goes in auth.config.ts which the proxy imports; the heavy Credentials provider that touches the DB goes in auth.ts.
Why JWT sessions instead of database sessions? Two reasons: the Credentials provider requires the JWT strategy, and JWT is stateless so there is no DB read per request to validate a session, which scales better.
How does a user's id get from the database into session.user.id? authorize() returns { id, ... } on success. The jwt callback copies user.id onto the token (baked into the signed cookie). The session callback copies token.id onto session.user.id. Any server component then reads it via await auth().
Why do we rely on a DB unique constraint for duplicate emails instead of checking first? A "does this email exist" check followed by an insert has a race condition: two requests could both pass the check and then both insert. The unique constraint (@unique) makes the database the single arbiter; we catch error code P2002 to report the duplicate.
Why cap password length at 72? bcrypt only considers the first 72 bytes. Without the cap, a longer password would be silently truncated, which is a subtle correctness/security bug.
What does the proxy matcher exclude and why? It excludes /api/auth, Next's internal static/image assets, the favicon, and any path with a file extension, so the auth gate only runs on real page routes, not on assets or the auth endpoints themselves.
Why protect a route in both the proxy and the layout? Defense in depth. The proxy redirects early; the layout's await auth() guard ensures the page still refuses to render for a logged-out user even if a matcher rule were misconfigured. The layout also provides the session for the header UI.
What changed in Prisma 7 that broke our first build? Two things: the connection URL moved out of schema.prisma into prisma.config.ts, and new PrismaClient() now requires a driver adapter argument. We pass new PrismaClient({ adapter: new PrismaPg({ connectionString }) }).
Why bcryptjs instead of native bcrypt? bcryptjs is pure JavaScript, so there is no native binary to compile or to break on a serverless platform like Vercel. Slightly slower, but far more robust to deploy.

25. Authorization - the crux of the whole phase top

This is the most important idea in Phase 3, and the thing interviewers deliberately test. Authentication proved who you are (Phase 2). Authorization makes sure you can only touch your own data.

The attack we are defending against

Every application has an id, and the id appears in the URL when you edit: /applications/app_a1/edit. A curious or malicious user could simply change that id in the address bar to someone else's application id and hit enter. If the code naively fetched "the application with this id", it would happily hand over a stranger's data. This class of bug even has a name: IDOR (Insecure Direct Object Reference). It is one of the most common real-world security holes.

How we make it impossible

The trick is simple and absolute: never look up a row by id alone. Always require it to also belong to the current user.

Reads

lib/data/applications.ts

export function getApplication(userId: string, id: string) {
  return prisma.application.findFirst({
    where: { id, userId },   # must match BOTH the id AND the owner
  });
}
Why findFirst and not findUnique

findUnique can only search by a single unique field (the id), so it cannot also require userId. findFirst lets us add the userId condition. If the application belongs to someone else, the where matches nothing and we get null - and the edit page then calls notFound(), showing a 404. The attacker cannot even tell whether the id exists.

Writes

lib/actions/applications.ts

await prisma.application.updateMany({
  where: { id, userId: user.id },   # same double condition
  data: parsed.data,
});
// deleteMany works the same way
await prisma.application.deleteMany({ where: { id, userId: user.id } });
Why updateMany / deleteMany instead of update / delete

update and delete also require a single unique field, so they cannot carry the userId guard. updateMany and deleteMany accept a full where. If the row is not owned by the user, they simply affect zero rows - no error, no leak, no change. We check the returned count and report "not found" if it was zero. This means even a perfectly forged request with a real id it does not own does absolutely nothing.

The one helper that ties it together

Every read and write starts with const user = await requireUser(). Because the user id always comes from the verified session (never from the form or URL), the attacker cannot supply their own userId. They are permanently locked to their own data.

We proved it, not just claimed it

During Phase 3 we created two users, A and B, gave A an application ("Acme Corp"), then checked with real HTTP requests:

  User A → GET /applications              → "Acme Corp" appears        (1 match)
  User A → GET /applications/app_a1/edit  → 200 OK
  User B → GET /applications              → "Acme Corp" absent         (0 matches)
  User B → GET /applications/app_a1/edit  → 404 Not Found
Interview gold

If asked "how do you stop a user seeing another user's data", answer with the exact mechanism: "I never fetch by id alone. Reads use findFirst with { id, userId }; writes use updateMany/deleteMany with the same guard so a mismatched owner affects zero rows. The userId always comes from the session, never the request. I verified it with two users." That is a senior answer.

26. Mutations and cache revalidation top

A mutation is any change to data (create, update, delete). Ours are Server Actions - the same pattern as the Phase 2 login, so you already know the shape.

lib/actions/applications.ts (create)

export async function createApplication(_prev, formData) {
  const user = await requireUser();               # 1. who is this?
  const parsed = parse(formData);                  # 2. validate + clean input
  if (!parsed.success) return { error: ... };       # 3. bounce bad input

  await prisma.application.create({
    data: { ...parsed.data, userId: user.id },       # 4. save, stamped with owner
  });

  revalidatePath("/applications");              # 5. refresh the cached list
  redirect("/applications");                    # 6. send the user back to it
}

What is revalidatePath and why do we need it?

Next.js caches rendered pages to make them fast. The applications list is one of those cached pages. If we add a new application but do not tell Next the list changed, the user could be shown the old, stale list. revalidatePath("/applications") says "the data behind this URL changed, throw away its cache and rebuild it next time." So the new row shows up immediately.

A simple analogy

Caching is like a printed menu. It is fast because you do not re-check the kitchen every time. But when a dish changes, you must reprint the menu or people order things that no longer exist. revalidatePath is the reprint.

The update and delete actions do the same three-step opening (who, validate, scope), then their scoped updateMany/deleteMany, then revalidatePath.

27. The CRUD user interface top

One form for both create and edit

components/applications/application-form.tsx is a single client component used by both the New and Edit pages. It takes three props: the action to run, optional defaultValues (empty for create, the existing row for edit), and a submitLabel. Writing one form instead of two means the fields, styling, and validation live in exactly one place.

How the edit form knows which application to save

The update action needs the application id, but we do not want the id to be an editable field in the form (a user could tamper with it). So we "pre-attach" the id to the action on the server before handing it to the form:

const action = updateApplication.bind(null, application.id);

.bind creates a new version of the function with the first argument (the id) already filled in. The form just calls action(prevState, formData) and never sees or controls the id.

Belt and suspenders

Even if someone bypassed bind and forged a different id, the updateMany({ where: { id, userId } }) guard from section 25 would still refuse to touch a row they do not own. The id being non-editable is convenience; the scoping is the real security.

The list page

app/(app)/applications/page.tsx is a server component. It reads the ?status= value from the URL, validates it against the allowed statuses, fetches the (scoped) list, and renders a table. Two details worth noting:

  • Filtering by URL: the status filter is just links to /applications?status=APPLIED and friends. The state lives in the URL, so a filtered view is shareable and survives a refresh - no client-side state needed.
  • Empty state: when there are no applications, we show a friendly "Add your first one" panel instead of a blank table. Real apps always handle the empty case; it is a small thing that signals care.
Gotcha: searchParams and params are async now

In Next 15/16, a page's searchParams and params are Promises. You must await them: const { status } = await searchParams;. Older tutorials access them synchronously, which no longer works.

Delete with a confirm

The delete button is a tiny client component wrapping a form whose action is the bound deleteApplication. It adds a browser confirm() dialog on submit, so a misclick does not silently destroy data. If the user cancels, we preventDefault() and nothing happens.

29. The second relationship: Application to Event top

Phase 4 adds the timeline: each application can carry a list of events (applied, online assessment, interview, follow-up, offer, rejected, or a plain note). This is the second one-to-many relationship in the app, and it stacks on the first:

  User  ──(1 to many)──▶  Application  ──(1 to many)──▶  Event
  one person             many job apps                many timeline
                                                      events per app

The Event table already existed in our schema from Phase 2 - we just had not used it yet. Each event row stores which application it belongs to (applicationId), a type (from a fixed enum), optional notes, and an optional scheduledAt date.

Reading an application together with its events

Prisma lets us fetch a row and its related rows in one query using include:

lib/data/applications.ts

export function getApplicationDetail(userId: string, id: string) {
  return prisma.application.findFirst({
    where: { id, userId },                       # still scoped to the owner
    include: {
      events: { orderBy: { createdAt: "desc" } },  # newest first
    },
  });
}
Why include instead of a second query

include fetches the application and its events in a single round-trip and returns them as one nested object (application.events). Writing a separate query for the events would mean two trips to the database and more code. This is the ORM earning its keep.

30. Transitive authorization top

In Phase 3 the rule was "a user can only touch applications where userId matches." Events are trickier: an event has no userId of its own. It belongs to a user through its application. So ownership is transitive: you own an event if you own the application it hangs off.

We enforce this two ways depending on the operation.

Adding an event: check the parent first

lib/actions/events.ts (addEvent)

const owned = await prisma.application.findFirst({
  where: { id: applicationId, userId: user.id },   # do you own the parent?
  select: { id: true },
});
if (!owned) return { error: "Application not found" };  # no -> refuse

await prisma.event.create({ data: { ...parsed.data, applicationId } });

We confirm the application belongs to the user before attaching an event to it. If they do not own it, we stop.

Deleting an event: a relation filter

Prisma can filter by a condition on a related table. That lets us delete an event only if its application is owned by the user, in one query:

lib/actions/events.ts (deleteEvent)

await prisma.event.deleteMany({
  where: { id: eventId, application: { userId: user.id } },  # event AND its app's owner
});
What that where clause means

application: { userId: user.id } is a relation filter. It reads as "the event whose id is eventId and whose parent application has this userId." If a user passes an event id that belongs to someone else's application, the filter matches nothing and, exactly like Phase 3, zero rows are deleted. Same guarantee, one relationship deeper.

Interview

Nested/child resources are where a lot of real apps leak data, because people forget the child has no owner column of its own. Being able to say "I scope child writes through a relation filter on the parent's userId" is a strong, specific answer. We proved it: user B got a 404 on user A's application detail, so B cannot even see A's events, let alone delete them.

33. Drag and drop top

Phase 5 turns the plain list into a kanban board: six columns (one per pipeline stage) with cards you can drag between them. Dropping a card in a new column changes that application's status. It is the same data as the list, shown as a pipeline you can manipulate directly.

Why a library, and which one

Drag and drop looks simple but is genuinely hard to get right: mouse and touch and keyboard input, scroll while dragging, accessibility, collision detection. We use @dnd-kit, the modern, actively maintained, accessible toolkit for React.

Alternatives rejected

react-beautiful-dnd is deprecated and unmaintained. The browser's built-in HTML5 drag and drop has poor touch support and awkward ergonomics. @dnd-kit supports pointer, touch, and keyboard dragging out of the box, which is the accessible, professional choice.

The three moving parts

Piece@dnd-kit toolIn our code
The arenaDndContextkanban-board.tsx wraps everything and receives the drag events
A draggable carduseDraggablekanban-card.tsx
A drop target (column)useDroppablekanban-column.tsx

Each card's draggable id is the application id. Each column's droppable id is the status string (WISHLIST, APPLIED, ...). So when a drag ends, "which card, dropped where" is literally "which application id, onto which status."

kanban-board.tsx (the drop handler)

function handleDragEnd(event) {
  const id = String(event.active.id);          # the card = application id
  const status = event.over?.id;                # the column = new status
  if (!status) return;                        # dropped outside any column
  const card = optimistic.find((a) => a.id === id);
  if (!card || card.status === status) return;  # no move, nothing to do
  startTransition(async () => {
    applyMove({ id, status });              # update UI instantly (section 35)
    await updateApplicationStatus(id, status);  # persist in the database
  });
}

The drag handle

Notice the card has a small grip icon in the corner, and only that icon carries the drag listeners. The rest of the card is a normal link to the detail page. This "drag handle" pattern avoids a classic conflict: if the whole card were draggable, clicking it to open the detail could accidentally start a drag. Separating the handle from the link makes both behaviors reliable.

34. Passing data from server to client top

Drag and drop needs interactivity, so the board must be a client component. But the applications live in the database, which only server components can read. We bridge the two the standard way:

  board/page.tsx  (server component)
    requireUser()  →  listApplications(user.id)   ← scoped read
    │  maps rows to { id, company, role, status }
    ▼  passes as a prop
  <KanbanBoard applications={cards} />   (client component)
    handles all the dragging in the browser
Why not fetch in the client

The server component already has a secure, user-scoped database connection. Fetching there and passing the result down means no extra API endpoint, no loading spinner on first paint, and the authorization stays on the server where it belongs. The client component only receives the already-filtered data it is allowed to see.

The board page passes only the four fields a card needs (id, company, role, status), not the whole row. Sending the minimum keeps the client bundle small and avoids leaking fields the UI does not use.

35. Optimistic updates top

When you drop a card in a new column, waiting for the database round-trip before the card visibly moves would feel laggy. So we move the card immediately in the UI, then save in the background. This is an optimistic update: we optimistically assume the save will succeed.

React gives us a hook built exactly for this: useOptimistic.

kanban-board.tsx

const [optimistic, applyMove] = useOptimistic(
  applications,                              # the real data from the server
  (state, move) =>                           # how to compute the temporary view
    state.map((a) => (a.id === move.id ? { ...a, status: move.status } : a)),
);

We render optimistic (not the raw prop). When a drag happens, applyMove instantly produces a version of the list with that card in its new column, so the UI updates the moment you drop. Meanwhile the server action saves the change and calls revalidatePath("/board"), which sends down fresh real data. Once that arrives, useOptimistic automatically discards the temporary view and shows the confirmed truth.

The safety net

If the save ever failed, useOptimistic would snap the card back to where it really is, because the optimistic view only lives as long as the pending action. The user is never shown a lie for longer than the request takes. This is the honest way to do "instant" UI.

Gotcha

Optimistic updates must be dispatched inside a transition (or a server action call). We wrap applyMove and the await updateApplicationStatus together in startTransition, so React ties the temporary state to the lifetime of that save.

Interview

"How do you make a drag feel instant without lying to the user?" Answer: optimistic UI with useOptimistic, tied to the server action inside a transition, so a failed save reverts automatically. The status update on the server is still a scoped updateMany, so the security story from Phase 3 carries straight over.

37. Deriving metrics from raw data top

Every phase so far stored or displayed rows. The dashboard is different: it computes new information that is not stored anywhere. This is the most "engineering" part of the app, because the value is in the calculation, not the storage.

All the math happens in one server-side function, getDashboardStats(userId). It reads the user's applications once, then reduces them into numbers.

lib/data/stats.ts (the shape)

const applications = await prisma.application.findMany({
  where: { userId },                       # still scoped to the owner
  select: { status: true, createdAt: true },   # only the columns the math needs
});

// count how many applications sit in each status
for (const app of applications) statusCounts[app.status] += 1;
Why compute on the server, and why select only two columns

Computing on the server means the raw rows never leave it - the browser receives only the finished numbers, which is smaller and does not leak data. Selecting just status and createdAt (the only fields the metrics use) keeps the query lean; there is no reason to pull company names or notes to count statuses.

Interview

"The dashboard aggregates in a single query and computes rates and a funnel in memory, all user-scoped." Aggregation and derived views are what separate a data-entry app from a product that gives insight - exactly the "computed views" an interviewer looks for.

38. The metrics, explained plainly top

Response rate

Of the jobs you actually applied to, how many did the employer respond to (in any way, including a rejection)?

  • Applied = every application except those still in Wishlist (Wishlist means you have not applied yet).
  • Responded = anything that moved to OA, Interview, Offer, or Rejected. A rejection is still a response.
  • Response rate = responded / applied, as a percentage.

Interview rate and offer rate follow the same idea (reached interview-or-beyond / applied, and offers / applied).

The pipeline funnel

A funnel shows how many applications reached each successive stage: Applied, then OA, then Interview, then Offer. Each step is a subset of the one before, so the bars shrink - that shrinking is where you "leak" candidates.

An honest limitation

We only store each application's current status, not its full history of stage changes. So the funnel is computed from the current stage: an application at Offer is counted as having reached Applied, OA, Interview, and Offer. This is why the chart is labelled "based on current stage." A fully accurate funnel would require reading the event timeline (which does record transitions), and we could switch to that later. Being upfront about this tradeoff is exactly the kind of judgment worth showing.

Applications over time

We bucket applications by the week they were added, for the last 8 weeks, so you can see your activity trend. The bucketing is plain JavaScript: find the Monday of each week, then count applications whose createdAt falls in that 7-day window.

function startOfWeek(date) {
  const d = new Date(date);
  const mondayOffset = (d.getDay() + 6) % 7;  # days since Monday
  d.setDate(d.getDate() - mondayOffset);   # rewind to Monday
  d.setHours(0, 0, 0, 0);
  return d;
}

41. One-click guest login top

A recruiter will spend about thirty seconds on your project. If they have to sign up before seeing anything, most will leave. So the single highest-value feature is a button that drops them straight into a fully populated app. That is the guest login.

The design: a fresh sandbox per click

Every time someone clicks "Try the demo", we create a brand new guest user with its own copy of sample data. So two visitors never step on each other, and anyone can drag, edit, and delete freely without affecting the real thing.

Why a fresh user instead of one shared demo account

A single shared account would let one visitor delete another's view mid-demo, and their edits would pile up. A fresh, isolated guest per click is predictable and safe. We mark these users with an isGuest flag so they are easy to identify and clean up later.

How it plugs into the existing auth

We add a second Credentials provider to NextAuth, with the id "guest". Unlike the email/password one, it takes no input - its authorize just creates the guest and returns it:

auth.ts

Credentials({
  id: "guest",
  credentials: {},
  async authorize() {
    const user = await createGuestUser();       # make user + seed data
    return { id: user.id, email: user.email, name: user.name, isGuest: true };
  },
}),

Because it is just another provider, everything downstream (JWT session, the proxy guard, per-user scoping) works exactly the same. The guest is a real, logged-in user - it simply has no password and is flagged.

Knowing a user is a guest

We thread the isGuest flag through the same JWT pipeline that carried the user id (section 13): authorize returns it, the jwt callback stores it on the token, the session callback exposes it as session.user.isGuest. The app layout reads that to show a small "you are exploring a demo" banner with a link to sign up.

42. Seeding demo data top

Seeding means filling a fresh account with believable starter content so the app does not look empty. Empty dashboards and blank boards make a great product look broken; seeded data makes it look alive.

createGuestUser() creates the user, then loops over a hand-written list of about ten realistic applications (Stripe, Vercel, Notion, and so on) spread across all six statuses, each with a few timeline events. It uses Prisma's nested create to make an application and its events in one call:

lib/guest.ts

await prisma.application.create({
  data: {
    ...fields,
    userId: user.id,
    events: { create: events },   # create the child events in the same call
  },
});
Two thoughtful details

Spread-out dates: each demo application's createdAt is set to a different point over the past several weeks, so the "applications over time" chart actually shows a trend instead of one tall bar. No real password: the guest's passwordHash is a sentinel string that can never match bcrypt.compare, so nobody can log into a guest account by guessing - it is only reachable through the guest button.

Interview

"The guest login provisions an isolated user and seeds it with date-spread demo data via a nested Prisma create, so the board and dashboard look real on first load." This shows product sense, not just coding - you thought about the person evaluating the app.

43. Public vs private routes, and polish top

Until now every page required login. Phase 7 adds a public landing page at / with the pitch, the demo button, and sign in / sign up links. That means the auth guard needs an exception.

Recall the authorized callback (section 13) that the proxy runs on every request. We add a small allow-list:

auth.config.ts

const PUBLIC_PAGES = ["/"];
// ...inside authorized():
if (PUBLIC_PAGES.includes(nextUrl.pathname)) return true;  # anyone may see the landing
return isLoggedIn;                                        # everything else still gated
