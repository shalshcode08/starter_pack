Phase 0Foundation and tooling

Getting a database, a way to talk to it, and the right tools in place before writing any features.

The big picture: what is this project, really?

Recall lets a student upload their own PDFs (lecture notes, textbook chapters) and then ask questions and get answers drawn only from those PDFs, with a pointer to the exact page an answer came from. This kind of system has a name: RAG, short for "Retrieval-Augmented Generation." The idea in one sentence: instead of trusting an AI to answer from memory (where it can make things up), you first retrieve the relevant passages from the user's own documents, then ask the AI to generate an answer using only those passages.

For any of that to work, we need three foundations before we write a single feature: a place to store data (a database), a translator between our code and that database, and a set of build tools. That is all Phase 0 is.

Foundation 1: the database (PostgreSQL on Neon)

A database is just an organized place to store information so you can find it again quickly. We use PostgreSQL ("Postgres"), a decades-old, extremely reliable database that professionals reach for by default.

But we do not run Postgres on our own computer or rent a server for it. We use Neon, a service that hosts a Postgres database for us in the cloud, for free. You give it a name, it gives you back a connection string - a single line of text that contains the address, username, and password all in one, like a full postal address for your database.

Why Neon specificallyTwo reasons. First, it has a genuinely free tier, and this whole project has a hard rule of costing Rs 0. Second, and more importantly, Neon has a feature called branching: just like code branches in Git, you can have a "development" copy of your database that is separate from "production." You are using the development branch's connection string right now, which means experiments here can never corrupt real user data later.

What is pgvector, and why did we turn it on?

Here is the one piece of magic that makes RAG possible. Computers cannot compare the meaning of two sentences directly. So we convert each passage of text into a long list of numbers - typically hundreds of them - called an embedding or a vector. The trick is that passages with similar meanings get similar lists of numbers. "The mitochondria produces energy" and "cells generate power in the mitochondria" end up close together in this number-space, even though they share few words.

To find passages relevant to a question, we turn the question into a vector too, then ask the database: "which stored vectors are closest to this one?" Ordinary Postgres cannot do that. The pgvector extension teaches it how - it adds a new column type for vectors and the math to compare them. We enabled it with one idempotent command (idempotent means safe to run repeatedly - it does nothing if already done):

CREATE EXTENSION IF NOT EXISTS vector;
NoteWe ran this early, as a standalone check, purely to confirm Neon's free tier actually supports pgvector before building anything on top of it. If that command had failed, the entire technical plan would have needed rethinking. It succeeded, so we knew the ground was solid. (Later, in Phase 1, this same line gets baked into a migration so it runs automatically on any fresh database.)

Foundation 2: the translator (Prisma)

Talking to a database traditionally means writing SQL, a specialized query language. It works, but it is easy to make mistakes that your editor cannot catch until the program crashes. Prisma is an ORM (Object-Relational Mapper): it lets us describe our data in a clean schema file, and then write database operations as normal TypeScript with autocomplete and type-checking. If you ask for a field that does not exist, your editor underlines it in red immediately, before the code ever runs.

The driver adapter: how Prisma actually connects

Prisma 7 (the version we use) connects to Postgres through a small, standard library called pg, wrapped in something called a driver adapter. You can think of the adapter as a power plug shape: pg is a well-understood, widely-used way of speaking to Postgres, and the adapter is the socket that lets Prisma plug into it.

Why the driver-adapter approachOlder Prisma shipped its own private connection engine written in a different programming language (Rust), bundled as a separate binary. That worked but was heavier and occasionally awkward to deploy. The modern adapter approach uses a plain JavaScript library instead, which is lighter, deploys more predictably on serverless platforms like Vercel, and is the direction Prisma is actively moving. Choosing it signals you are current with the ecosystem - which matters for a project meant to showcase engineering skill.

Where the connection string lives (and why not in the schema)

We keep the connection string - which contains a password - in a file called .env, which is never committed to Git. The code reads it by name (DATABASE_URL), never by pasting the secret directly. In Prisma 7 this wiring lives in prisma.config.ts:

import "dotenv/config";
import { defineConfig } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: { path: "prisma/migrations" },
  datasource: {
    url: process.env["DATABASE_URL"],  // read the secret from .env, never hard-code it
  },
});
GotchaA subtle Prisma 7 change bit us here and is worth remembering. In older Prisma, the connection URL went inside schema.prisma itself. In Prisma 7 that is no longer allowed - the URL moves to this config file, and the schema's datasource block only names the database type. When we briefly tried the old way, the editor flagged it instantly: "the datasource property url is no longer supported." That is the type-checking safety net doing its job.

Foundation 3: the tools (pnpm, tsx, dotenv)

  • pnpm is a package manager - the tool that downloads and organizes the third-party code libraries our project depends on. It does the same job as npm or yarn but is faster and more disk-efficient. This project standardizes on it exclusively, so mixing in npm would create conflicting bookkeeping files.
  • tsx lets us run a TypeScript file directly from the command line, without a separate compile step. Handy for quick one-off maintenance scripts.
  • dotenv is the small library that reads the .env file and loads those secrets into the running program.

Proving the foundation holds

A foundation you have not tested is just a hope. We confirmed the connection genuinely works by running a trivial query against Neon and watching it succeed. The result: Postgres reachable, pgvector available, Prisma configured. Only then did we move on.

Phase 1The data model

Describing every kind of information Recall stores, and how the pieces connect.

What a data model is, in plain terms

A data model is a blueprint of everything the app needs to remember and how those things relate. Think of it as a set of labeled boxes and the string connecting them. Recall's boxes:

  • User - a person with an account.
  • StudySet - a folder grouping documents by topic, e.g. "DBMS Finals." Each belongs to one user.
  • Document - one uploaded PDF. Each belongs to one study set.
  • Chunk - a small passage of text cut out of a document, plus its embedding vector. Each belongs to one document. This is the heart of the whole system.
  • Conversation and Message - the chat history of questions and answers.
  • Citation - a link recording exactly which chunk an answer drew from, so we can show the source.
  • Question - a generated quiz item or flashcard, each tied back to the chunk it came from.

The connections form a strict hierarchy: User -> StudySet -> Document -> Chunk. This is not decoration - it is how we guarantee privacy. Because every chunk traces up through a document and a study set to exactly one user, we can always answer "who is allowed to see this?" with certainty.

Why chunks, instead of storing whole documents?

A textbook chapter might be 30 pages. If a student asks "what is a foreign key?", we do not want to hand the entire chapter to the AI - it is wasteful and dilutes the relevant part. So during upload we slice each document into small, page-sized passages (chunks). We embed each chunk separately. Then a question retrieves just the handful of chunks that actually discuss foreign keys. Chunks are the unit of retrieval, the unit of citation, and the unit of quiz-generation - everything downstream is built on them.

Expressing the model in Prisma

Each box becomes a model in the schema. Here is the Chunk model, the most important one, annotated:

model Chunk {
  id         String                       @id @default(cuid())
  documentId String
  content    String
  embedding  Unsupported("vector(768)")?
  page       Int
  chunkIndex Int
  tokenCount Int

  document   Document   @relation(fields: [documentId], references: [id], onDelete: Cascade)
  citations  Citation[]
  questions  Question[]

  @@index([documentId])
}
Line by line
  • id ... @default(cuid()) - every chunk gets a unique ID, auto-generated. We use cuid (a random-looking collision-resistant string) rather than simple counting numbers 1, 2, 3.
  • content - the actual passage text.
  • embedding Unsupported("vector(768)")? - the meaning-vector. More on this just below. The ? means it can be temporarily empty (a chunk exists briefly after being cut but before being embedded).
  • page, chunkIndex - where in the document this passage sits, so a citation can say "page 12."
  • tokenCount - roughly how long the passage is, useful for staying within the AI's size limits later.
  • document ... onDelete: Cascade - the string tying this chunk to its parent document. "Cascade" means: if the document is deleted, its chunks are automatically deleted too, so we never leave orphaned data behind.
  • @@index([documentId]) - a lookup shortcut so "give me all chunks of this document" is fast.

The tricky part: the vector column

Prisma does not natively understand pgvector's special vector type - it is too new and specialized. Prisma gives us an escape hatch for exactly this situation: Unsupported("..."). It means "there is a real database column here of this exact type; I, Prisma, will keep track that it exists and include it in migrations, but I will not let you read or write it through the normal typed API." We will read and write embeddings through small hand-written SQL queries in later phases, which is the standard, accepted pattern for pgvector.

Why 768 numbers, fixed foreverThe vector's length must match whatever AI model produces the embeddings. We use Google's gemini-embedding-001, configured to output exactly 768 numbers. This choice is essentially permanent: if we ever changed it, every stored embedding would become incompatible and every document would need re-processing from scratch. So we committed to 768 deliberately and up front, rather than discovering the constraint later.
Road not taken
Keeping the vector column out of the schema entirely. We could have hidden the embedding from Prisma completely and managed it with pure hand-written SQL. Rejected because then the schema file would lie about the shape of the table - anyone reading it would not know embeddings exist. Using Unsupported keeps the schema honest and self-documenting.
Road not taken
Simple counting IDs (1, 2, 3...) instead of cuid. Rejected for two reasons: sequential IDs leak information (a competitor could see "Document 5" and guess you only have five documents), and they make it awkward to generate an ID before saving. Random cuids avoid both problems and are the professional default.
Road not taken
The ivfflat vector index instead of HNSW. pgvector offers two ways to make similarity search fast. We chose HNSW (explained next), which gives better speed-and-accuracy at the cost of more memory - a worthwhile trade for a quality-focused project.

Turning the blueprint into real tables: migrations

A migration is a recorded, replayable set of database changes - the SQL instructions that build the tables. Prisma can generate these automatically by comparing our schema to the current database. But we needed two things Prisma's auto-generation does not know about: turning on pgvector, and creating the special vector index. So we generated the migration without applying it and then hand-edited it:

pnpm prisma migrate dev --create-only --name init

The --create-only flag means "write the SQL file but do not run it yet," giving us a chance to add our two lines. At the very top we added the extension, so a brand-new database (like the future production branch) sets itself up correctly:

-- Enable pgvector before any vector column is created
CREATE EXTENSION IF NOT EXISTS vector;
Why order mattersThe Chunk table has a vector(768) column, and Postgres only knows what "vector" means after the extension is enabled. Putting this line first guarantees the type exists before any table tries to use it. This is why the migration must be self-contained: it should rebuild the entire database from nothing, in the right order, without any manual steps.

At the bottom, we added the index that makes similarity search fast:

-- HNSW index for cosine similarity search over chunk embeddings
CREATE INDEX "Chunk_embedding_hnsw_idx" ON "Chunk" USING hnsw ("embedding" vector_cosine_ops);
What this index does, and why "cosine"Without an index, finding the closest vectors means comparing the question against every single chunk - fine for ten chunks, unbearably slow for a hundred thousand. HNSW (Hierarchical Navigable Small World) builds a clever multi-layer map of the vectors so search can leap to the neighborhood of good answers in a few hops instead of scanning everything. vector_cosine_ops tells it to measure closeness by cosine similarity - essentially, the angle between two vectors rather than the raw distance. Cosine is the standard choice for text embeddings because it focuses on direction (meaning) and ignores magnitude (roughly, passage length).

With the file edited, we applied it for real:

pnpm prisma migrate dev

This ran our SQL against Neon, creating all eight tables, their relationships, their indexes, and the vector index in one shot - and recorded that this migration has been applied, so it never runs twice.

The connection singleton

Our app needs one shared Prisma client to talk to the database. We create it once in lib/db.ts and import it everywhere:

import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "@/app/generated/prisma/client";

const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL });

const globalForPrisma = globalThis as unknown as { prisma?: PrismaClient };

export const prisma =
  globalForPrisma.prisma ?? new PrismaClient({ adapter });

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
Why the global danceThe odd-looking globalForPrisma trick solves a real, specific problem. During development, Next.js constantly reloads your code as you edit. If we naively wrote new PrismaClient(), every reload would open a fresh batch of database connections and never close the old ones, quickly exhausting the free tier's connection limit. So we stash the one client on the global object and reuse it across reloads. In production this is unnecessary (code does not hot-reload there), so we only do it when not in production. This exact pattern is near-universal in professional Next.js + Prisma codebases.

Proving Phase 1 holds

Same principle as before: test the foundation. We ran a quick script that created a user with a nested study set, counted the users, then deleted everything and confirmed the cascade cleaned up - all green. That proved the tables, relationships, cascades, and the generated typed client genuinely work together against the live database. Phase 1 stands.

Phase 2Authentication and authorization

Letting people prove who they are, and making sure they only ever see their own data.

What "auth" actually means here

Two related but different jobs hide under the word "auth":

  • Authentication - proving who you are. In Recall this is an email and a password. Get them right and the system believes you are you.
  • Authorization - deciding what you are allowed to do. In Recall the rule is simple and absolute: you may only ever touch data that belongs to you. Your study sets, your documents, your chats - never anyone else's.

These matter more here than in a typical app because Recall holds a person's private study material. A leak is not a cosmetic bug, it is a breach of trust. So authorization is not a feature we bolt on later - it is a rule every single database query in the app must obey.

We build all of this on Auth.js (formerly NextAuth), the standard authentication library for Next.js. It handles the fiddly, security-critical parts - sessions, cookies, cross-site request protection - so we do not hand-roll them and get them subtly wrong.

A scope decisionThe original plan allowed signing in with Google too. We deliberately cut that to email and password only. It removes an external dependency (no Google Cloud Console setup, no OAuth round-trip to debug) and, as you will see next, actually makes the whole design simpler and cleaner. The plan file was updated to match - the plan follows reality, not the other way around.

The central decision: how a session is remembered

After you prove who you are, the app needs to remember that for the rest of your visit, so you are not asked to log in on every click. That memory is called a session, and there are two ways to store it:

  • Database sessions - the server keeps a row in a table for every logged-in visit and hands you a meaningless ticket number (a cookie). On each request it looks up your ticket to see who you are.
  • JWT sessions - the server hands you a small, cryptographically signed token that itself says "this is user X." No database lookup needed; the signature proves it has not been tampered with. JWT stands for JSON Web Token.

We chose JWT sessions, and here the choice was partly made for us. Auth.js has a hard rule: the email-and-password login method (its "Credentials" provider) only works with JWT sessions, not database sessions. Since email/password is our only login method, JWT is the natural fit.

The happy consequenceBecause we use JWT and no external login providers, we do not need Auth.js's database adapter at all - and therefore none of the extra bookkeeping tables (Account, Session, VerificationToken) it would normally require. Our User table stays exactly as the PRD drew it: id, email, passwordHash, name, createdAt. This is why we deliberately left those adapter tables out of Phase 1 - their necessity depended on a decision we had not made yet. Cutting Google collapsed the whole thing to its simplest honest form.
Road not taken
Database sessions with a full adapter. More moving parts and, critically, incompatible with password login. Rejected - it would have forced us to keep Google or drop password login, and added three tables we do not need.
Road not taken
Rolling our own auth from scratch. Tempting to feel "in control," but authentication is exactly the kind of code where small mistakes become security holes. Using a well-audited library is the professional choice; reinventing it is the amateur one.
Road not taken
Storing passwords encrypted (reversible) instead of hashed. A serious error. Encryption can be undone with a key; if that key leaks, every password leaks. Hashing (next section) is one-way by design. We never store anything that can be turned back into the original password.

Never store the password: hashing with bcrypt

We must never save a user's actual password. If our database were ever stolen, plain passwords would be a catastrophe - people reuse passwords across sites. Instead we store a hash: the output of a one-way mathematical blender. The same password always produces the same hash, but there is no way to run the blender backwards to recover the password.

When you sign up, we hash your password and store only the hash. When you sign in, we hash what you typed and compare the two hashes. We use bcrypt, a hashing function designed specifically for passwords - it is deliberately slow, which makes it painfully expensive for an attacker to guess billions of passwords, while staying fast enough for a single honest login.

// On sign-up: turn the password into a one-way hash (the "12" sets the work factor)
const passwordHash = await bcrypt.hash(password, 12);
await prisma.user.create({ data: { name, email, passwordHash } });

// On sign-in: compare, never decrypt
const valid = await bcrypt.compare(password, user.passwordHash);
The work factorThat 12 is the number of rounds of blending, on a doubling scale. Higher means slower and safer; too high and honest logins feel sluggish. Twelve is the current well-accepted default - strong today, and easy to raise later as computers get faster.

A subtlety: the edge/node split

Next.js can run little bits of code in two very different places. Most code runs on a normal server ("Node"), where powerful libraries like bcrypt work fine. But some code - specifically middleware, the guard that inspects every incoming request before it reaches a page - runs in a stripped-down, ultra-fast environment called "the edge," where bcrypt cannot run.

Our middleware needs to answer one cheap question on every request: "is this person logged in, and are they allowed here?" It does not need bcrypt for that. So we split the Auth.js configuration in two:

  • auth.config.ts - the lightweight, edge-safe part: which pages are protected, how to read the session token. No bcrypt, no database.
  • auth.ts - the full part that adds the password-checking login method (which needs bcrypt and the database), built by extending the lightweight config.

The middleware imports only the light half, so it stays edge-compatible; the actual login imports the full half. Here is the guard logic that decides who may go where:

authorized({ auth, request: { nextUrl } }) {
  const isLoggedIn = !!auth?.user;

  // Logged-in users skip the login/register pages
  const isOnAuthPage = nextUrl.pathname.startsWith("/login")
                    || nextUrl.pathname.startsWith("/register");
  if (isOnAuthPage) {
    if (isLoggedIn) return Response.redirect(new URL("/dashboard", nextUrl));
    return true;
  }

  // The app's private areas require a session
  const isProtected = nextUrl.pathname.startsWith("/dashboard")
                   || nextUrl.pathname.startsWith("/study-sets");
  if (isProtected) return isLoggedIn;

  return true;  // everything else (landing page, etc.) is public
}
Why guard in middleware at allBecause it runs before the page does. An unauthenticated visitor to /dashboard is bounced to /login before any private component even starts rendering. We confirmed this works: a raw request to /dashboard returned a redirect to /login, exactly as intended. (Middleware is the outer fence; the real, non-negotiable checks still happen at each database query, covered below.)

The sign-up flow, using Server Actions

Rather than a separate hand-written API endpoint, we use Server Actions - a modern Next.js feature where a form can call a server function directly. The form on the page points at a function that runs safely on the server, and the framework wires up the plumbing between them. It is less code and harder to misuse.

export async function registerUser(_prev, formData) {
  // 1. Validate the shape of the input with a schema (zod)
  const parsed = registerSchema.safeParse({
    name: formData.get("name"),
    email: formData.get("email"),
    password: formData.get("password"),
  });
  if (!parsed.success) return { error: parsed.error.issues[0]?.message };

  // 2. Refuse duplicate accounts
  const existing = await prisma.user.findUnique({ where: { email } });
  if (existing) return { error: "An account with this email already exists" };

  // 3. Hash, store, and sign the new user straight in
  const passwordHash = await bcrypt.hash(password, 12);
  await prisma.user.create({ data: { name, email, passwordHash } });
  await signIn("credentials", { email, password, redirectTo: "/dashboard" });
}
Validate before you trustStep 1 uses zod, a library that checks incoming data against a declared shape ("email must look like an email, password at least 8 characters"). Never trust what a browser sends - it could come from a script, not a form. We validate in two places: instantly in the browser for friendly feedback, and again on the server, which is the check that actually counts.

We verified this data path against the live database: registering hashes and stores the user, a duplicate email is caught, the correct password verifies, and a wrong password is rejected. All four behaved correctly.

The real safeguard: enforcing ownership at every query

Middleware is a fence around the front door, but the true guarantee lives deeper. Every feature from here on will fetch data through a single identity helper, so there is one obvious place the ownership rule is enforced:

export async function requireUser() {
  const session = await auth();
  if (!session?.user?.id) redirect("/login");
  return session.user;   // guaranteed to have an id from here on
}
Why funnel through one helperBecause scattered security checks are where breaches hide - miss one query and you have a leak. By making requireUser() the mandatory first step of every protected page and action, and then always filtering queries by that user's id (for example, "find this study set where userId equals the current user"), we make the safe path the easy path. Later phases lean on this constantly, especially the vector search - every similarity query will be scoped to the owner, so one student can never retrieve another's notes.

Proving Phase 2 holds

Four independent checks, all green: the project type-checks; the middleware redirects an unauthenticated request away from /dashboard; the login and register pages render; and the credential logic (hash, duplicate guard, correct-vs-wrong password) behaves correctly against the live database. The password-login backbone is in place, and every later feature now has a trustworthy notion of "the current user" to build on.

Phase 3Study sets

The folders that organize a student's documents - and the first place the app starts to feel real.

What we are building

A study set is a labelled folder, like "DBMS Finals" or "Organic Chemistry Ch. 4." It groups the PDFs a student wants to study together, and later it is the boundary for a conversation and for generated quizzes. Everything in Recall hangs off study sets, so this phase gives users the ability to create, list, rename, and delete them. Programmers call this basic set of operations CRUD (Create, Read, Update, Delete).

It sounds mundane, and the happy path is. The interesting engineering is entirely in one question: how do we make it impossible for one user to touch another user's study sets? That is what most of this phase is really about.

The three layers

Rather than cram database calls, security checks, and HTML into one file, we separate the work into three clean layers. This is how experienced developers keep code testable and safe as it grows.

  • Data-access layer (lib/data/study-sets.ts) - the only code that talks to the database about study sets. Every function here takes a userId and refuses to look outside it.
  • Action layer (lib/actions/study-sets.ts) - the bridge the browser calls. It figures out who is asking (via requireUser()), validates the input, then delegates to the data layer.
  • UI layer (the pages and components) - what the user sees and clicks. It knows nothing about the database directly; it only calls actions.
Why bother separating themBecause security lives in the data layer, and by funnelling every study-set database call through those few functions, there is exactly one place to get the ownership rule right - and one place to audit. Scatter raw database queries across a dozen page files and sooner or later one of them forgets the ownership check. That forgotten check is the leak. Centralizing prevents it structurally, not just by being careful.

The key idea: authorization lives inside the query

The naive way to check ownership is two steps: first fetch the study set, then compare its userId to the current user in your code. That works, but it is fragile - forget the second step and you have a hole. We do something stronger: we bake the ownership rule into the database query itself, so a query physically cannot touch a row that is not yours.

// Rename, scoped by BOTH id and userId at once:
export function renameStudySet(userId: string, id: string, title: string) {
  return prisma.studySet.updateMany({
    where: { id, userId },      // must match the id AND belong to this user
    data: { title },
  });
}
Read the trick carefullyNotice this uses updateMany, not the simpler update. The reason is subtle and important. update targets a single row by its id and throws an error if you add other conditions it cannot guarantee - and more to the point, it would let you attempt an update by id alone. updateMany takes a full where filter, so we require id and userId to both match. If an attacker passes someone else's study-set id, the filter matches zero rows: the update simply affects nothing and reports count: 0. No error to handle, no leak - the wrong-owner case collapses to "nothing happened." Deletes use the same deleteMany pattern for the same reason.

We tested exactly this attack. We created a study set owned by one user, then had a second "attacker" user attempt to read, rename, and delete it. Every attempt came back empty: read returned nothing, rename and delete each affected 0 rows. The owner's own operations, meanwhile, worked normally. The isolation is real and enforced at the deepest layer.

Server Actions, and keeping the screen fresh

As in Phase 2, the browser talks to the server through Server Actions - server functions a form can call directly. The create action shows the full shape:

export async function createStudySetAction(_prev, formData) {
  const user = await requireUser();                       // 1. who is asking?
  const parsed = studySetTitleSchema.safeParse(formData.get("title"));
  if (!parsed.success) return { error: ... };             // 2. is the input valid?
  const set = await createStudySet(user.id, parsed.data); // 3. do it, scoped to them
  revalidatePath("/dashboard");                           // 4. refresh the list
  redirect(`/study-sets/${set.id}`);                      // 5. go to the new set
}
What revalidatePath doesNext.js caches rendered pages for speed. When you create a study set, the dashboard's list is now stale - it does not show the new one. revalidatePath("/dashboard") tells Next "the data behind this page changed, rebuild it next time it is shown." Without it, a user would create a set and not see it appear until a manual refresh. This is the modern alternative to the old approach of manually re-fetching data in the browser - the server just marks what is stale and the framework handles the rest.

The UI layer stays thin: the dashboard server component calls listStudySets(user.id) and renders either an empty state or a grid of cards; a small client component handles the create form's typing and pending state; the detail page offers inline rename and a delete button guarded by a confirmation prompt.

A real bug we caught (and why testing matters)

While verifying this phase, the app logged a cryptic MissingSecret error from the auth library. The cause was almost comically small: back in Phase 2, when we added the secret key to the .env file, the existing line had no newline at its end, so the new AUTH_SECRET value got glued onto the end of the database line. The file looked fine at a glance, but the tool that reads it saw one mangled line and silently ignored the secret.

The lessonThis bug would not have shown up in the compiler or in casual clicking around - the redirect for logged-out users still worked, because it needs no secret. It would have struck the instant a real user tried to actually log in. We only caught it because we drove the real login flow end to end and watched the server logs. "It compiles" and "the page loads" are not the same as "it works." Splitting the two .env values onto separate lines fixed it immediately.

Proving Phase 3 holds

Several independent checks, all green. The project type-checks. We drove a complete real login (obtaining a security token, signing in through the auth library's own endpoint, receiving a session cookie, and using it to load the protected dashboard, which rendered the signed-in user's email). The study-set operations work for their owner. And the cross-user attack - read, rename, delete another user's set - failed at every turn, exactly as designed. The organizing layer of the app is in place and safe, ready for documents to be poured into it.

Phase 4PDF ingestion

Turning an uploaded PDF into searchable, embedded chunks - the engine room of the whole product.

This is the most involved phase, and the one that makes Recall more than a note organizer. The goal: a student uploads a PDF, and the app turns it into a form that can be searched by meaning and later answered from. Everything here builds toward that.

Embeddings, properly this time

Back in Phase 0 we described an embedding as turning text into a list of numbers that captures its meaning. Now we actually use them, so it is worth understanding a little deeper. An embedding is a point in a space with many dimensions - ours has 768. The remarkable property, learned by a large AI model from vast amounts of text, is that meaning becomes geometry: passages about similar ideas land near each other, passages about different ideas land far apart. "How do I stop the database returning made-up rows?" and "foreign keys enforce referential integrity" are worded completely differently, yet their points sit close together, because they are about the same thing.

We generate embeddings with Google's gemini-embedding-001 model, on its free tier. Two refinements we deliberately added:

  • Task types. The model can tailor an embedding to its purpose. When embedding a chunk from a document we tell it RETRIEVAL_DOCUMENT; when embedding a user's question we say RETRIEVAL_QUERY. This asymmetry is designed for exactly our use case - matching questions to passages - and measurably improves the matches.
  • Normalization. We scale every vector to a length of one (called L2-normalizing). Google recommends this whenever you use fewer than the model's full dimensions, and it makes the distance math clean and consistent.
Why 768 and not the model's defaultThis model can emit up to 3072 numbers per embedding. We asked it for 768 instead. Fewer numbers means less storage and faster search, and it lets us keep the exact database column we designed back in Phase 1 - no migration, no re-thinking the index. The model supports this truncation natively (a design called Matryoshka, where shorter vectors are still meaningful), so we lose very little quality for a real efficiency gain.

The core constraint: the serverless timeout

Here is the problem that shapes this entire phase. Recall runs on Vercel's free tier, where each web request must finish within a short time budget (roughly ten seconds). But embedding a large PDF is slow - a 40-page chapter might become 150 chunks, each needing a call to Google's model. Do that all inside the single upload request and it will run past the limit and be killed, leaving a half-processed document. The PRD names this explicitly: ingestion must be incremental, never blocking on one long request.

The design: split the work, poll for progress

We break ingestion into two stages with very different costs:

  1. Upload (one quick request). The PDF arrives; we extract its text, slice it into chunks, and save all of them to the database at once - marking the document processing. This part touches no AI model, so it is fast. Crucially, we then throw the PDF file away - we only ever needed the text.
  2. Process (many tiny requests). A separate endpoint embeds just 20 chunks per call and returns immediately. The browser calls it over and over - a loop - each call chewing through another 20 chunks, until none remain. Then the document flips to ready.

Because no single request ever embeds more than 20 chunks, no request comes close to the timeout, no matter how large the PDF is. A 1000-chunk textbook simply means more loop iterations, not a longer request. The browser shows a live progress bar the whole time.

// The browser's loop (simplified): keep asking the server to embed the next batch
let remaining = totalChunks;
while (remaining > 0) {
  const { status, remaining: left } = await postJSON(`/api/documents/${id}/process`);
  remaining = left;
  updateProgressBar(totalChunks - remaining, totalChunks);
  if (status === "ready") break;
}
How a request knows what is left to doEach /process call asks the database a simple question: "give me up to 20 chunks belonging to this document that do not yet have an embedding." It embeds those, saves the vectors, and reports how many still lack one. There is no shared memory or job queue tracking progress - the database itself is the progress tracker. This is what makes it robust: if a call fails or the user reloads, the next call just picks up wherever the unfinished chunks are. State lives in the one place that survives everything.

Reading the PDF and cutting it up

Extracting text from a PDF is genuinely hard - PDFs describe visual layout, not clean prose - so we lean on a library, unpdf, to pull the words out. We ask it for text page by page rather than as one blob, because we need to remember which page each piece came from. That page number is what lets a future answer say "see page 12" - the citation feature depends on capturing it here.

Then we chunk: cut each page into passages of roughly 500 tokens (a token is about four characters), breaking at sentence boundaries so we never slice a sentence in half. Consecutive chunks share a small overlap - the tail of one chunk repeats at the head of the next.

Why overlap, and why this sizeSize is a balance: too large and a chunk covers several topics, blurring its meaning into mush; too small and it lacks enough context to be meaningful. Around 500 tokens is a well-tested sweet spot. The overlap solves a subtler problem - if an important idea happens to straddle a chunk boundary, without overlap it might be split so badly that neither half captures it. Repeating a little text across the seam means no idea falls through the cracks.

Storing the vectors (where Prisma steps aside)

Recall from Phase 1 that our friendly database toolkit, Prisma, does not understand the special vector type - we marked that column Unsupported. So while the chunks' text is saved through Prisma normally, writing and reading the embeddings uses small hand-written SQL queries. Selecting the next batch to embed is a raw query for chunks "where embedding is null"; saving a vector is a raw update. It is a deliberate, contained exception - everywhere else we keep the type-safe toolkit.

-- Find the next unembedded batch (Prisma cannot express "embedding IS NULL")
SELECT id, content FROM "Chunk"
WHERE "documentId" = $1 AND embedding IS NULL
ORDER BY "chunkIndex" ASC LIMIT 20;

-- Save one chunk's embedding (the ::vector cast turns our text into a real vector)
UPDATE "Chunk" SET embedding = $1::vector WHERE id = $2;
Road not taken
Storing the original PDF file. We discard it after extracting text. Keeping it would need a paid file-storage service (against the zero-cost rule) and buys us nothing - citations point to page numbers, not to the file, and we never need to show the original again. Less stored, less to secure, no dependency.
Road not taken
A background job queue (the "proper" enterprise way). The textbook solution to slow work is a worker queue running separately from web requests. It is genuinely better at scale - but every option costs money or adds a service to run. The polling-loop design achieves the same "never block, always resumable" guarantee using only the tools we already have. Choosing the simpler thing that fully meets the requirement, over the fancier thing that does not fit the constraints, is itself the engineering.
Road not taken
Embedding all chunks in one request. The obvious first attempt, and exactly what the timeout forbids. Rejected by the constraint, not by preference.

A tooling snag worth remembering

After adding the new libraries, the development server suddenly refused to start. The cause was a security feature of our package manager: some packages ship little setup scripts that run at install time, and modern pnpm refuses to run them until you explicitly approve each one - and until you decide, it treats the situation as unresolved and fails. Our two new packages were sitting in that undecided state.

The fix, and the judgment behind itWe had already proven the packages work without their setup scripts (the ingestion test used them successfully). So rather than blindly allow the scripts to run, we explicitly told pnpm "do not run these, and stop asking" - the safer choice, since it avoids executing third-party install code we do not need. The lesson mirrors Phase 3's: modern tools have safety gates that block until you make a conscious decision, and the right move is to understand what the gate is protecting against before waving it through.

Proving Phase 4 holds

We verified the entire pipeline against the real Google model and the real database, end to end. Starting from a PDF built with known study text, the system extracted both pages, cut them into chunks (keeping the key sentence intact), embedded every chunk, and stored the vectors. Then the real test of the whole idea: we asked, in completely different words, "What guarantees referential integrity in a database?" - and the closest stored passage was exactly the one about foreign keys enforcing referential integrity. Meaning matched meaning, across different wording, scoped to the owning user. The upload and process endpoints also correctly refuse anyone without a session. The retrieval engine works - which is precisely what Phase 5 will turn into grounded, cited answers.

Phase 5Grounded Q&A

The payoff: ask a question, get an answer built only from your own notes - with citations, and an honest "I don't know" when the notes don't cover it.

Everything so far has been foundation. This phase is the product. A student types a question; Recall finds the relevant passages in their own material, hands them to an AI model with strict instructions, and streams back an answer that points to the exact page it came from. This is the full RAG loop - Retrieval-Augmented Generation - finally assembled.

Retrieval: finding the right passages

When a question arrives, the first job is to find the handful of chunks most likely to answer it. We embed the question into the same 768-dimension space as the chunks (this time telling the model the text is a RETRIEVAL_QUERY, its query-optimized mode), then ask the database for the nearest stored chunks by cosine distance - the HNSW index from Phase 1 making this fast.

SELECT c.content, c.page, d.title,
       (c.embedding <=> $1::vector) AS distance   -- <=> is cosine distance
FROM "Chunk" c
JOIN "Document" d ON d.id = c."documentId"
JOIN "StudySet" s ON s.id = d."studySetId"
WHERE s."userId" = $2 AND s.id = $3               -- scoped to THIS user and set
ORDER BY c.embedding <=> $1::vector
LIMIT 6;
The two lines that guarantee privacyLook at the WHERE clause. Every retrieval is filtered by the current user's id and the specific study set. It is impossible for this query to return a chunk belonging to anyone else, or even to the same user's other subjects. The PRD's rule that "every vector query is authorization-scoped to the user" is not a policy we remember to follow - it is welded into the SQL. This is the single most important line in the whole feature.

Refusing to make things up

A generic chatbot, asked something your notes do not cover, will cheerfully invent a plausible-sounding answer. For a study tool that is worse than useless - it teaches you wrong things with false confidence. Recall must instead say "that isn't in your notes." We achieve this with two layers of defense, and the story of why we need two is instructive.

The obvious approach is a distance threshold: if even the closest chunk is too far from the question, refuse without even asking the AI. We built that. But when we tested it, a totally unrelated question ("who won the 1998 football world cup?") landed at distance 0.54 from our database notes - under a loose threshold. The lesson: the embedding space is compressed, so "clearly unrelated" is not as far away as intuition suggests. A distance cutoff alone is a blunt instrument.

Two-tier refusalSo we use the distance guard only as a coarse filter - it catches the hopeless cases and saves an unnecessary AI call. The fine judgment is delegated to the model itself: we instruct it to answer only from the provided passages and to explicitly declare when they do not contain the answer. When we gave the football question's (weakly-related) passages to the model, it correctly replied: "The provided sources do not contain information about the 1998 football world cup." The model, reading the actual text, is a far better judge of relevance than a single distance number. Belt and braces.

Grounding the model

The retrieved chunks are assembled into a prompt, each numbered and labelled with its document and page. Alongside it goes a firm system instruction - the model's standing orders:

Answer ONLY using the numbered sources provided. Never use outside knowledge.
If the sources do not contain enough information, say so plainly. Do not guess.
On the final line, list the source numbers you used, like: SOURCES: 1, 3
If you could not answer, the final line must be: SOURCES: none

The passages themselves are formatted so the model knows exactly where each came from:

[1] (from "DBMS Notes", page 4)
A foreign key enforces referential integrity between two tables...

[2] (from "DBMS Notes", page 7)
Normalization organizes tables to reduce redundancy...
Why "only from these sources" is the whole gameThis single constraint is what separates Recall from a chatbot that hallucinates. By forbidding outside knowledge and forcing the model to work only from retrieved text, every answer is anchored to the student's real material. We also set a low "temperature" (0.2), which makes the model favour faithful, predictable phrasing over creative flourishes - exactly what you want when accuracy matters more than style.

Citations you can trust

An answer is only trustworthy if you can check it. So the model ends every response with a hidden bookkeeping line - SOURCES: 1, 3 - naming which passages it actually used. The server reads that line, maps those numbers back to the exact chunks (which carry a document title and page), and records them as Citation rows linked to the answer. The line itself is stripped out so the user never sees the plumbing; instead they see tidy source chips under the answer: "DBMS Notes - p.4".

Why a trailer line instead of inline tagsWe considered having the model sprinkle markers like [1] through its prose, but parsing those reliably out of a live-streaming response is fiddly and error-prone. A single, predictable line at the very end is trivial to parse with confidence, and it cleanly separates "the answer the human reads" from "the machine-readable record of sources." Each citation traces to a specific chunk, so the promise that every answer is checkable against its exact origin holds.

Streaming the answer

Waiting in silence for a paragraph to generate feels broken. So the answer is streamed: the model emits words as it produces them, the server forwards each fragment immediately, and the browser paints them as they arrive - the familiar live-typing effect. The moment the stream ends, the browser quietly refreshes from the database, so the polished, citation-carrying version rendered by the server replaces the raw streamed text.

Two representations, on purposeDuring streaming the browser shows the raw text for immediacy (hiding the trailing SOURCES: line as it goes). But the authoritative version comes from the database after the answer is saved, complete with its citation chips. This split keeps the live experience snappy while making the stored record - what you see on reload - the single source of truth. The database is always the last word.
Road not taken
Letting the model use its own knowledge to "fill gaps." This is what makes chatbots untrustworthy for study. Explicitly forbidden - an answer that is not from the student's notes is a bug here, not a feature.
Road not taken
A distance threshold as the only refusal mechanism. Tested and found too blunt (the football question slipped under it). Kept as a coarse pre-filter, but the model's own reading of the passages is the real judge.
Road not taken
Multiple separate conversations per study set. More UI and state for little v1 value. One conversation per study set keeps the experience focused; threaded chats can come later.

Proving Phase 5 holds

Verified end to end against the live model and database. A relevant question - worded completely differently from the source ("what keeps a database from referencing missing records?") - returned a correct, grounded answer and a citation pointing to the exact foreign-key passage on its page. An irrelevant question was refused, the model plainly stating the notes did not cover it, with no citations attached. Driving the real web endpoint over the network produced the same streamed answer, and afterwards the database held the question, the answer (with its bookkeeping line stripped), and a citation resolving to "DB Notes, page 1." Unauthenticated requests are rejected. The heart of the product beats correctly: it answers from your notes, shows its work, and admits when it does not know.

Phase 6Quiz and flashcard generation

Turning a study set into practice material - questions and flashcards, each traceable to the exact passage it came from.

Reading notes is passive; testing yourself is how things stick. This phase generates two kinds of active-recall material from a study set: quiz questions (a question and its answer) and flashcards (a term and its explanation). Both are built the same way - sample the study set's chunks, ask the model to write items, and store each one linked to its source.

Structured output: making the AI return data, not prose

For chat, we wanted flowing text. Here we want the opposite: clean, machine-readable data - a list of items, each with a prompt, an answer, and a pointer to its source. If we just asked the model in plain English and parsed the reply, a single stray word or missing bracket could break everything. Instead we use a feature called structured output: we hand the model a strict schema describing the exact shape we want, and the API guarantees the response conforms to it.

// The shape we demand back - an array of items, each with these fields
{
  type: ARRAY,
  items: {
    type: OBJECT,
    properties: {
      sourceIndex: { type: INTEGER },   // which source this came from
      prompt:      { type: STRING },     // the question or term
      answer:      { type: STRING },     // the answer or definition
    },
    required: ["sourceIndex", "prompt", "answer"],
  },
}
Why a schema beats "please return JSON"Politely asking a model for JSON usually works - until the one time it wraps the output in an explanation, or trails off, and your code crashes on real users. Declaring the schema moves the guarantee from hope to contract: the response is valid structured data by construction. This is the professional way to get data (rather than prose) out of a language model, and it makes the whole feature reliable instead of flaky.

Tracing every item back to its source

The PRD insists each generated item be "traceable to where it came from." We achieve this the same way as citations in Phase 5: the chunks are numbered when handed to the model, and each item it returns names the sourceIndex it drew from. Back in our code, we map that number to the real chunk's id and store it on the question as sourceChunkId. So every flashcard and quiz question is permanently linked to the exact passage that justifies it - and the interface shows that source (document and page) right on the card.

A small guard that mattersModels occasionally cite a source number that does not exist, or return a blank field. Before saving, we discard any item whose sourceIndex does not map to a real chunk, or whose prompt or answer is empty. Only well-formed, genuinely-sourced items reach the database. Trusting model output blindly is how bad data creeps in; a cheap validation pass keeps the study material honest.

Pressing "Generate" replaces the previous set of that type, so a student always gets a fresh batch rather than an ever-growing pile. Quiz cards reveal their answer on click; flashcards flip between term and explanation - simple, familiar study interactions.

Proving Phase 6 holds

Verified against the live model using the seeded sample chapter. Generation produced eight quiz questions and eight flashcards; every item mapped to a real source chunk from the material; and after saving, reading them back showed each carrying its source document and page. The generate-and-replace flow, the structured schema, and the source-linking all work together correctly.

Phase 7The guest demo

Letting anyone try the real product in one click, with zero signup and zero risk to other users' data.

A portfolio project nobody can try is a screenshot. The guest demo removes every barrier: one click and you are inside a working account, pre-loaded with a sample chapter, free to ask questions and generate quizzes immediately. The engineering challenge is doing this safely.

The isolation problem

The tempting shortcut is a single shared "demo account" that everyone logs into. It is easy - and it is a trap. Two strangers using it at once would see each other's questions, overwrite each other's generated quizzes, and generally trip over one another. Worse, anyone could delete the sample material and break the demo for the next visitor. A shared mutable account is a bad experience waiting to happen.

So instead, every guest gets their own private, throwaway account, pre-filled with a personal copy of the demo material. Guests are completely isolated from each other and from real users - the same authorization rules from Phase 3 apply, no exceptions for being a guest.

Cloning the sample - without paying to re-embed

Here is the neat part. The sample chapter is seeded once into a hidden template account - its text extracted, chunked, and embedded (the expensive, slow steps from Phase 4). When someone clicks "try the demo," we do not re-run any of that. We clone the template's content into the new guest account, copying the chunks and their existing embeddings straight across in the database.

-- Copy every chunk of a template document into the guest's new document,
-- embeddings and all, in one statement. No AI calls, no waiting.
INSERT INTO "Chunk" (id, "documentId", content, embedding, page, "chunkIndex", "tokenCount")
SELECT gen_random_uuid()::text, $1, content, embedding, page, "chunkIndex", "tokenCount"
FROM "Chunk" WHERE "documentId" = $2;
Why copy embeddings instead of regeneratingEmbedding a chapter costs time and API quota. Every guest re-embedding the same fixed sample would be pure waste - slow for the user (a spinner on the one action that must feel instant) and needless load on the free-tier model. Since the sample never changes, its vectors are computed once and copied byte for byte. A guest account materializes in a moment, fully searchable. This is the kind of efficiency instinct - do expensive work once, reuse the result - that separates thoughtful engineering from the naive path.

A second front door

Recall's normal way in is email and password. For guests we added a second login method to the auth system - a "guest" provider with no form to fill in. When triggered, instead of checking a password it runs the clone-and-create routine and hands back a brand-new guest identity. The rest of the app cannot tell the difference: a guest has a normal session and is subject to all the same rules. We tag these accounts with an isGuest flag, which powers a small "Guest mode" badge in the header and lets a routine cleanup script sweep away old guest accounts so they do not pile up forever.

Reusing the machinery, not bypassing itIt would have been easy to hack in a special back door that skips authentication for guests. We deliberately did not. By expressing "guest" as just another login provider inside the same auth system, guests flow through the exact same session handling, the same middleware, and the same per-user data scoping as everyone else. The safe design is the one where there are no special cases to get wrong.

Proving Phase 7 holds

Verified end to end. Triggering a guest login created a fresh isolated account whose email marks it a guest, pre-loaded with its own copy of the Photosynthesis sample - all twelve chunks present with their embeddings intact. A meaning-based search on that cloned data correctly found the passage about the carbon-fixing enzyme, proving the copied vectors work. Driving the real one-click flow over the network landed on a dashboard showing the "Guest mode" badge and the sample study set, ready to use. And on that same cloned material, quiz and flashcard generation produced properly-sourced items. A stranger can now go from link to asking questions of real AI in a single click, touching no one else's data.

Phase 8Polish, deployment, and documentation

Turning a working app into something a recruiter or user can understand, try, and trust.

By Phase 7, Recall already had the hard product machinery: auth, uploads, embeddings, retrieval, citations, generated study material, and an isolated guest demo. Phase 8 is about the last mile. A visitor should not land on a starter template. A small phone should not make forms feel broken. A future reviewer should be able to read the README and understand the architecture without spelunking through the code.

Responsive polish

The app already had useful empty, loading, and error states: document upload shows progress, chat disables itself until a PDF is ready, study tools explain when generation is unavailable, and forms display server-side errors. The main polish gap was layout resilience. Several controls assumed a single horizontal row, which is fine on a laptop but cramped on a phone.

<form
  ref={formRef}
  action={formAction}

>
  <input className="min-w-0 rounded-lg ..." />

This pattern says: on small screens, stack the input and button; on wider screens, use a two-column row. The min-w-0 detail matters because CSS grid and flex layouts otherwise let long text refuse to shrink, causing overflow. We applied the same idea to the app header, study set title controls, upload button, chat form, and study tools toolbar.

Road not takenWe did not redesign the whole authenticated app. The existing interface is intentionally quiet and work-focused, which fits a study tool. Phase 8 improves robustness and presentation without burying the actual product under a decorative rewrite.

Build reliability

The first production build failed for an environmental reason: the app used next/font/google, which tries to fetch font files from Google during the build. In a restricted or offline environment, that download fails even though the product code is fine. A portfolio app should be easy to build anywhere, so we removed that external font dependency and used system fonts.

export default function RootLayout({ children }) {
  return (
    <html lang="en" className="h-full antialiased">
      <body className="min-h-full flex flex-col">{children}</body>
    </html>
  );
}
Why this is not just cosmeticA web font is visual polish, but a reliable build is more important. Removing the network dependency means local builds, CI builds, and Vercel builds are less fragile. The app still looks professional, and it no longer needs a font download to compile.
